# Mono 独立开发指导书

> 写给项目作者。
>
> 当你不知道下一步该开发什么、要不要增加一个模块、或者该如何参考 Pi 与
> Oh My Pi 时，先回到这份文档。

## 1. 你正在做什么

Mono 的长期目标是一个能够日常使用的 Coding Agent。

它应当能够：

1. 理解一个代码仓库。
2. 根据用户需求修改代码。
3. 使用工具验证修改结果。
4. 在执行过程中接受审批、中断和转向。
5. 保存会话，并让失败过程可以检查和恢复。

你的开发方式是：

> 保持一个可运行的 Coding Agent 主链路，再围绕真实问题逐步加固各个模块。

这意味着你既不需要一次设计完整个 Harness，也不应该把所有章节模块分别写完后
才尝试集成。

## 2. 当前状态

截至 2026-06-12，Mono 已经完成第一块地基：Runtime。

现有能力包括：

- 统一块消息模型
- Provider 流式适配
- Agent Loop
- 工具执行与参数校验
- 运行时事件
- Token 预算与上下文压缩
- 审批、中断、暂停和转向
- JSONL 会话转录
- 基础监督策略
- ExecutionEnv 副作用能力边界与工作区约束

当前验证基线：

```text
python -m pytest -q
80 passed
```

Runtime 现在负责完成一轮轮推理和工具执行。它不应该继续吸收会话产品、项目规则、
复杂工具或用户界面的职责。

## 3. 目标架构

Mono 接下来沿四层演进：

```text
CLI / 未来的 RPC 或其他界面
            |
            v
CodingAgentSession
面向用户的 Coding Agent 使用方式
            |
            v
HarnessSession
会话生命周期、配置、上下文、策略和持久化
            |
            v
Runtime
模型推理、工具调用、控制和运行事件
            |
            v
Provider / Tool / Storage
外部能力适配器
```

每层只解决自己的问题：

| 层 | 负责什么 | 不负责什么 |
|---|---|---|
| Runtime | 执行一次 Agent Loop | 项目规则、产品工作流、复杂会话管理 |
| HarnessSession | 管理一次长期会话 | Coding Agent 专属工具体验 |
| CodingAgentSession | 组织代码修改和验证流程 | Provider 协议和底层执行细节 |
| CLI | 收集输入、展示事件和审批 | 决定运行时业务规则 |

当你准备增加功能时，先判断它属于哪一层。如果无法判断，说明功能边界还没有想清楚。

## 4. 开发原则

### 4.1 先做纵向链路

每个阶段都必须让用户获得一种新的完整能力。

合格的开发切片：

```text
用户要求修改一个函数
-> Agent 搜索并读取代码
-> Agent 修改文件
-> Agent 运行测试
-> Agent 汇报结果
```

不合格的开发切片：

```text
实现 Memory 接口
实现 MCP 接口
实现 Planner 接口
但主链路没有使用它们
```

### 4.2 模块因真实问题而产生

遇到下面的情况时再抽取模块：

- 一段职责已经有两个调用者。
- 一个组件难以独立测试。
- 修改一处行为需要同时改动多个无关文件。
- 当前主链路因为职责混杂而难以理解。

不要因为 Pi 或 OMP 中存在某个目录，就在 Mono 中创建同名目录。

### 4.3 每项能力都要有完成标准

“代码写完”不是完成。

一项能力完成时，你应当能够回答：

- 用户能通过哪条真实链路使用它？
- 正常情况如何验证？
- 失败、超时、取消和恢复如何验证？
- 它是否改变了已有模块的职责？
- 文档描述是否和实际代码一致？

### 4.4 保持主循环可读

Runtime、HarnessSession 和 CodingAgentSession 都会拥有自己的控制流。

每个控制流都应当能够按顺序讲清楚。不要让组件互相回调并修改对方内部状态。
需要跨层协作时，使用显式方法、返回值、事件或命令。

## 5. 开发路线

### 阶段一：HarnessSession

这是你现在应该开始的阶段。

### 目标

在 Runtime 之上增加稳定的会话生命周期，让 CLI 不再直接拼装和持有所有运行时组件。

### 需要解决的问题

- 谁创建并持有 Runtime、Provider、工具注册表和控制平面？
- 完整会话历史与模型本轮使用的压缩上下文如何分离？
- 恢复会话时如何恢复预算和运行配置？
- 同一个会话如何避免同时运行两个 prompt？
- 修改模型、工具和配置时，当前 Turn 与下一 Turn 分别看到什么？
- 会话发生错误后，调用者如何知道最终状态？

### 建议提供的最小接口

接口名称可以调整，但职责需要保持清楚：

```python
session = HarnessSession(...)

async for event in session.prompt("修复这个问题"):
    ...

session.steer("先不要修改配置文件")
session.abort()
session.status()
HarnessSession.resume(session_id)
```

### 暂时不做

- Session 分叉和树形历史
- 多用户并发
- 数据库后端
- 远程 RPC
- 插件系统

### 完成标准

- CLI 只依赖 HarnessSession，不再直接组装 AgentLoop。
- 完整转录和推理上下文拥有明确的不同职责。
- 恢复后的预算与消息状态正确。
- 同一会话无法并发执行两个 prompt。
- Runtime 的现有测试继续通过。
- HarnessSession 有单元测试和一条端到端测试。

### 阶段二：Coding Tools V1

### 目标

让 Mono 拥有完成小型代码修改所需的最小工具集。

### 第一批工具

| 工具 | 最低能力 |
|---|---|
| `read` | 按路径和行范围读取，限制最大输出 |
| `search` | 搜索文件名和文件内容 |
| `edit` | 精确修改文件，旧内容不匹配时拒绝写入 |
| `shell` | 在工作区内执行命令，支持审批、超时和中断 |

### 每个工具都要回答

- 输入 Schema 是否足够明确？
- 路径是否限制在工作区内？
- 输出过长时如何处理？
- 中断后是否还会继续产生副作用？
- 失败信息能否帮助模型修正下一步？
- 哪些操作必须审批？

### 暂时不做

- AST 编辑
- LSP
- 浏览器
- GitHub
- MCP 工具
- 后台任务

### 完成标准

Mono 能在一个测试仓库内完成：

```text
读取需求 -> 定位代码 -> 修改文件 -> 运行测试 -> 汇报结果
```

它不能修改工作区外文件，也不能绕过危险命令审批。

### 阶段三：Coding Agent MVP

### 目标

把 HarnessSession、Coding Tools 和 CLI 组合成一个你愿意实际使用的最小 Coding Agent。

### 需要加入的能力

- 根据当前工作区构建系统提示。
- 读取项目级规则文件。
- 在最终回答前运行必要验证。
- 清楚展示工具调用、错误、审批和最终状态。
- 会话退出后可以恢复。

### 此阶段最重要的问题

持续追问：

> Mono 为什么没有完成这个真实编码任务？

每次失败都应归类：

| 失败类型 | 优先检查 |
|---|---|
| 找不到代码 | search/read 工具和提示 |
| 修改失败 | edit 契约和上下文信息 |
| 不运行测试 | 系统提示和完成条件 |
| 重复调用工具 | 工具反馈和监督策略 |
| 会话越来越差 | 上下文构建和压缩 |
| 执行危险操作 | 权限策略和工作区边界 |

### 完成标准

Mono 能稳定完成一组小型真实编码任务，而不是只在 MockProvider 剧本中运行。

### 阶段四：真实任务回归集

### 目标

用可重复任务衡量 Mono，而不是凭使用感受判断进度。

### 第一批任务

准备几个小型 fixture 仓库，覆盖：

1. 修改一个函数并通过已有测试。
2. 定位并修复简单 Bug。
3. 添加一个小功能和对应测试。
4. 根据失败测试继续修复。
5. 拒绝修改工作区外文件。
6. 危险命令要求审批。
7. 中断后停止工具和文件修改。
8. 恢复会话后继续任务。

每个任务记录：

- 是否完成
- 使用的 Turn 和工具调用数量
- Token 消耗
- 出现的错误
- 是否需要人工纠正

当一次改动让旧任务退化时，先修复退化，再继续扩展。

### 阶段五：按问题增加高级能力

只有 Coding Agent MVP 和回归集稳定后，才进入这一阶段。

建议顺序：

1. **Skills / 项目指令**：解决重复工作方法无法复用的问题。
2. **扩展钩子**：解决用户需要定制行为的问题。
3. **MCP**：解决外部工具接入成本高的问题。
4. **长期 Memory**：解决跨会话需要保留项目知识的问题。
5. **Subagent**：解决单 Agent 无法有效处理可并行的大任务。
6. **RPC / ACP / Web UI**：解决终端以外的接入需求。

每项高级能力都需要一个真实失败案例作为开发理由。没有失败案例时，先不做。

## 6. 如何借鉴 Pi 和 Oh My Pi

### 6.1 Pi 是主要架构参考

阅读 Pi 时，重点理解职责分层和契约，不要逐文件翻译。

推荐阅读顺序：

1. `D:\Pi\pi\packages\agent\src\agent-loop.ts`
   - 看最小 Agent Loop 如何推进 Turn 和工具调用。
2. `D:\Pi\pi\packages\agent\src\agent.ts`
   - 看有状态 Agent 如何管理队列、中断和订阅者。
3. `D:\Pi\pi\packages\agent\docs\agent-harness.md`
   - 看 Harness 层为什么需要配置快照、操作锁和持久化边界。
4. `D:\Pi\pi\packages\agent\src\harness\agent-harness.ts`
   - 带着一个具体问题阅读，不要尝试一次看懂全部代码。
5. `D:\Pi\pi\packages\coding-agent\src\core`
   - 看通用 Harness 如何被包装成 Coding Agent 产品。

每次参考 Pi 时，写下四句话：

```text
Pi 在解决什么问题？
Mono 当前是否已经遇到这个问题？
Pi 的方案依赖了哪些前提？
Mono 最小需要借鉴哪一部分？
```

如果第二个问题的答案是否定的，暂时不要实现。

### 6.2 OMP 是成熟问题案例库

OMP 适合帮助你发现成熟 Coding Agent 会遇到哪些问题，例如：

- 工具质量和工具输出设计
- Subagent 隔离
- LSP、DAP 和编辑工具
- 评测、遥测与回归测试
- 多种产品入口和复杂扩展能力

不要把 OMP 的目录结构当成 Mono 的目标结构。OMP 已经包含大量产品历史、性能优化和
兼容负担。你当前需要研究它为什么增加某项能力，而不是复制能力本身。

## 7. 每次开发一个能力时怎么做

使用下面的循环：

### 第一步：写清真实任务

```text
作为 Mono 用户，我希望它能够 ______。
目前它会在 ______ 失败。
完成后，我可以通过 ______ 验证。
```

### 第二步：追踪现有主链路

从用户入口开始，追到失败位置。确认问题属于哪个模块，不要先创建新模块。

### 第三步：定义契约

先决定：

- 输入是什么？
- 输出是什么？
- 哪些错误属于正常结果？
- 哪些错误需要终止？
- 是否允许取消、重试和恢复？

### 第四步：先写失败测试

测试应描述用户可观察行为。避免只测试内部实现步骤。

### 第五步：做最小实现

只实现让真实任务和契约测试通过所需的内容。

### 第六步：运行完整验证

至少执行：

```powershell
python -m pytest -q
python -m compileall -q core runtime providers tools cli tests
git diff --check
```

当 Coding Agent 回归集建立后，也必须运行受影响的真实任务。

### 第七步：更新文档

记录：

- 新能力属于哪一层。
- 它改变了什么契约。
- 它刻意不解决什么。
- 后续触发什么问题时才需要继续扩展。

## 8. 当你不知道下一步做什么

按顺序检查：

1. 当前主链路是否有失败测试？
2. Coding Agent 是否有一个真实任务无法完成？
3. 最近的失败是否重复出现？
4. 失败属于工具、上下文、会话、运行时还是交互层？
5. 修复该失败所需的最小能力是什么？

如果没有真实失败，就运行回归任务或亲自使用 Mono 完成一个小型编码工作。

不要通过浏览 Pi 或 OMP 的功能列表来决定 Mono 的下一步。

## 9. 防止项目变乱的检查线

出现以下信号时，暂停新功能开发并整理结构：

- 同一种消息存在多套含义接近但无法转换的模型。
- CLI 开始直接修改 Runtime 内部状态。
- 一个模块同时负责控制流、存储和展示。
- 新功能需要修改五个以上不相关模块。
- 测试只能通过大量 Mock 内部细节完成。
- 文档写着“支持”，但主链路没有调用。
- 发生错误后无法判断状态是否已经提交。
- 恢复后的行为与恢复前不一致。

整理结构时只解决已经出现的问题，不做全局重写。

## 10. 当前行动清单

你现在从 HarnessSession 开始。

第一轮工作只需要完成：

- [ ] 写出 HarnessSession 的职责和不负责事项。
- [ ] 画出 `CLI -> HarnessSession -> Runtime` 的调用链。
- [ ] 定义 `prompt / steer / abort / status / resume` 的行为契约。
- [ ] 分离完整转录与本轮推理上下文。
- [ ] 增加同一会话的运行锁。
- [ ] 让现有 CLI 改为通过 HarnessSession 运行。
- [ ] 添加恢复、并发调用和端到端 prompt 测试。
- [ ] 用真实 Provider 完成一次最小会话。

完成以上内容后，再进入 Coding Tools V1。

## 11. 你暂时不需要解决的问题

在 Coding Agent MVP 跑通前，保持以下内容在范围之外：

- 多 Agent 编排
- MCP
- 长期记忆
- 插件市场
- LSP 和 DAP
- Web UI
- 自动 Git 工作流
- 多租户服务
- 分布式执行

这些能力以后可能有价值。现在加入它们只会让你更难判断基础链路为什么失败。

---

你衡量进度的方式很简单：

```text
Mono 今天比昨天多稳定完成了一类真实编码任务吗？
```

如果答案是肯定的，你就在正确方向上。如果答案是否定的，先查清正在开发的能力与
真实任务之间缺少哪条连接。
