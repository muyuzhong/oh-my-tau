# Lion Code 上下文管理基准测试说明

## 这份测试想回答什么

简历里的“降低 token 成本”至少包含两个不同问题：

1. **上下文本身是否变短**：比较累计输入 token 和单次请求峰值 token。
2. **实际计费是否下降**：缓存命中的 token 仍然存在，只是单价更低，因此要按缓存命中、缓存未命中和输出分别计费。

本测试同时报告两组数据，避免把“token 数减少”和“缓存折价”混成一个数字。

## 对照组

- `raw`：保存完整工具结果，不执行持久化、预算裁剪、过期清理和摘要；服务端自身的自动前缀缓存仍然生效。这是保守且公平的真实 API 基线。
- `managed`：使用当前项目的超大结果持久化、动态预算、过期结果清理、五分钟空闲清理和 85% 全量摘要逻辑。
- `eager_ablation`：只在“热缓存渐进式压缩”场景运行。它保留其他逻辑，但移除 60%～75% 区间的热缓存保护，用于观察“急着改写旧前缀”会损失多少缓存命中。这是机制消融实验，不是生产配置。

每条回放都使用独立且稳定的系统前缀，避免前一个场景污染后一个场景；同一条回放内部则逐轮追加消息，让服务端缓存可以像真实长会话一样复用旧前缀。

## 测试集如何构造

测试集定义在 [`dataset.json`](./dataset.json)。脚本读取当前仓库中的真实源码，再按指定字节数循环展开。这样既能制造跨阈值的长上下文，又不会在 Git 中提交数 MB 的重复文本。

四个场景分别是：

| 场景 | 真实素材 | 主要验证点 |
|---|---|---|
| 超大文件读取 | `agent.py`、`tools.py` | 30 KiB 持久化与预览截断 |
| 热缓存渐进式压缩 | `agent.py`、`prompt.py`、`memory.py` | 50%/60%/75% 阈值和热前缀保护 |
| 冷缓存过期结果清理 | `agent.py`、`autonomy.py`、`tools.py` | 模拟空闲 301 秒后的旧结果清理 |
| 全量摘要与事实保留 | `README.md`、`agent.py`、`prompt.py` | 85% 摘要、摘要成本和事实保留率 |

每个场景还放入 3～4 个容易核验的事实，例如路径、阈值和唯一代号。最后要求模型用 JSON 复述这些事实，脚本按精确字符串匹配计算保留率。

完整长文本由以下模板生成：

```text
[场景 <id>；第 <n> 轮；来源 <file>]
<当前仓库源码>
<当前仓库源码重复，直到达到 dataset.json 指定的字节数>
```

因此测试内容可读、可追溯，也能在仓库变化后重新生成。结果 JSON 会记录每个源文件的 SHA-256，明确本次数据对应哪个源码快照。

## 为什么按 180K 有效窗口测试

图片显示 `deepseek-v4-flash` 支持 1M 上下文，但当前项目的 `MODEL_CONTEXT` 没有该模型，未知模型会回退到 200K，并预留 20K 输出空间，所以压缩逻辑实际使用 **180K 有效窗口**。

本测试优先测量“当前代码现在真正会怎么运行”，不擅自把窗口改成 1M。报告会把这一点列为限制；如果后续把模型窗口登记为 1M，应重新运行基准，不能直接沿用本次百分比。

## 费用口径

按照用户提供的价格截图：

- 缓存命中输入：0.02 元 / 百万 token
- 缓存未命中输入：1 元 / 百万 token
- 输出：2 元 / 百万 token

实际费用计算公式：

```text
输入成本 = 缓存命中 token / 1,000,000 × 0.02
         + 缓存未命中 token / 1,000,000 × 1.00
总成本   = 输入成本 + 输出 token / 1,000,000 × 2.00
```

项目现有 `/cost` 使用另一组统一硬编码价格，因此本基准不使用 `/cost` 的金额。

## 如何运行

离线检查不需要 API：

```powershell
python benchmarks/context_management/benchmark.py --offline
```

在线测试需要在当前 PowerShell 进程设置密钥。不要把密钥写进脚本、结果或提交：

```powershell
$env:OPENAI_API_KEY = "<你的新密钥>"
python benchmarks/context_management/benchmark.py --online `
  --base-url "https://api.deepseek.com" `
  --model "deepseek-v4-flash" `
  --budget-cny 7.8
```

脚本在每次请求前进行保守预算检查；结果写入 `results/latest.json` 和 `REPORT_CN.md`。报告中的 API 消耗不包含手工连通性测试，因此总预算还应预留少量余量。

## 如何解释最终数字

优先用于简历的是：

- `raw` 与 `managed` 的累计输入 token 差异；
- 两组都享受服务端自动缓存时的实际输入成本差异；
- 峰值上下文和超过 180K 有效预算的请求次数；
- 事实保留率。

不要把 `eager_ablation` 当成主要基线，也不要把本次 180K 配置下的结果外推成 1M 配置的结果。
